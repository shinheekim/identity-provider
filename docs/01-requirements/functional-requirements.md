# 기능 요구사항 (Functional Requirements)

## 1. 개요
본 문서는 인증 서버(Auth Server)의 기능 요구사항을 정의한다.  
모든 인증 기능은 JWT 기반으로 동작하며, Access Token과 Refresh Token을 사용한다.

---

## 2. 인증 기능
### 2.1 회원가입

- 사용자는 ID와 Password를 이용하여 회원가입할 수 있다
- 회원가입 시 사용자 식별 정보(Identity)를 생성한다
- 동일한 ID(또는 이메일)는 중복 가입할 수 없다
- 비밀번호는 암호화하여 저장해야 한다
- 회원가입 완료 후 로그인 가능 상태가 되어야 한다
- 필요 시 회원가입 시점의 기본 상태값(활성/비활성)을 부여할 수 있다

---

### 2.2 일반 로그인

- 사용자는 ID와 Password를 통해 로그인할 수 있다
- 로그인 성공 시 Access Token과 Refresh Token을 발급한다
- 로그인 실패 시 적절한 에러를 반환한다
  - 단 일반 로그인시 비밀번호가 틀렸거나 사용자 아이디가 없는 경우 유추가 불가능하게 동일한 에러를 반환한다.
- 비밀번호는 암호화되어 저장되어야 한다

---

### 2.3 소셜 로그인

- 사용자는 외부 플랫폼을 통해 로그인할 수 있다
- 지원 플랫폼:
    - 카카오
    - 구글
- 소셜 로그인 성공 시 내부 Identity와 매핑된다
- 최초 소셜 로그인 시 회원가입 처리 또는 계정 연동 처리를 수행한다
- 이미 가입된 계정이 존재하는 경우 정책에 따라 기존 계정과 연결할 수 있다

---

### 2.4 로그아웃

- 사용자는 로그아웃할 수 있다
- 로그아웃 시 Refresh Token을 무효화한다
- 필요 시 Access Token 블랙리스트 처리 가능

---

### 2.5 토큰 발급

- 로그인 성공 시 다음 토큰을 발급한다
    - Access Token (짧은 만료 시간)
    - Refresh Token (긴 만료 시간)
- 토큰에는 사용자 식별 정보가 포함되어야 한다

---

### 2.6 토큰 검증

- 모든 보호된 API 요청은 Access Token을 통해 인증한다
- 서버는 JWT 서명을 검증하여 사용자 인증을 수행한다
- 토큰이 만료되었거나 유효하지 않은 경우 요청을 거부한다

---

### 2.7 토큰 재발급

- Access Token 만료 시 Refresh Token을 통해 재발급할 수 있다
- Refresh Token은 서버(DB 또는 Redis)에 저장되어야 한다
- 유효하지 않은 Refresh Token으로 요청 시 재발급을 거부한다

---

## 3. 계정 관리

### 3.1 Identity 관리

- 사용자는 하나의 Identity를 가진다
- 하나의 Identity는 여러 로그인 방식과 연결될 수 있다
    - 일반 로그인
    - 소셜 로그인

---

### 3.2 소셜 계정 연동

- 하나의 Identity에 여러 소셜 계정을 연결할 수 있다
- 동일한 소셜 계정은 하나의 Identity에만 연결된다

---

## 4. 비밀번호 관리

### 4.1 비밀번호 저장

- 비밀번호는 암호화하여 저장한다
- 평문 저장은 금지한다

---

### 4.2 비밀번호 변경

- 사용자는 비밀번호를 변경할 수 있다
- 기존 비밀번호와 동일한 비밀번호 사용 제한 가능
- 비밀번호 변경 이력을 관리할 수 있다

---

## 5. 로그인 이력

### 5.1 로그인 기록

- 로그인 시 이력을 저장한다
- 저장 항목:
    - 사용자 ID
    - 로그인 시간
    - 로그인 방식 (일반 / 소셜)
    - IP 주소
    - 성공 여부

---

## 6. 보안 정책

### 6.1 인증 정책

- Access Token은 짧은 만료 시간을 가진다
- Refresh Token은 별도로 관리된다

---

### 6.2 확장 고려

- 로그인 실패 횟수 제한 (계정 잠금)
- IP 기반 접근 제한
- 토큰 블랙리스트
- 서비스별 로그인 아이디 분리 
  - (통합 로그인 서버)
